# Release 0.2.0 Implementation Plan

Implementation plan for the items in [release-0.2.0-plan.md](release-0.2.0-plan.md). This document records decisions, rationale, and concrete implementation steps.

## Decision Log

Decisions made for this release, with rationale:

| Decision | Choice | Rationale |
|----------|--------|-----------|
| **People section name** (de-christianify) | "People in Mind" | Shorter, neutral, avoids prayer-related wording that may trigger Chinese app store censoring. "People I'm Thinking Of" is also valid but longer. |
| **Bible Notes replacement** | "Reading Notes" | Simple, neutral, matches the tone of "Reflections." "Notes from Reading" is redundant. Keeps the section useful for spiritual or secular reading without religious framing. |
| **Reminders** | Defer to 0.3.0 | Release 0.2.0 is already substantial (fixes, chip delete, localization, external API). Reminders add UserNotifications, settings UI, and persistence. Better to ship 0.2.0 and tackle reminders as a focused follow-up. |
| **Monthly calendar view** | Exploration doc only | Document pros/cons in `05-exploration-calendar-view.md`; implement in 0.3.0+ if the analysis supports it. Avoids scope creep and allows time to evaluate UX tradeoffs (accessibility, empty-cell semantics). |
| **Chinese localization** | zh-Hans only | Simplified Chinese covers the largest Chinese-speaking audience and primary censoring concerns. zh-Hant (Traditional) can be added in a later release if needed. |
| **External summarizer API** | Include in 0.2.0 | Release plan noted this may accelerate. The `Summarizer` protocol is already injectable; adding a cloud option differentiates the product and enables better labels for complex sentences. Use OpenAI-compatible API (https://chat.cloudapi.vip) with NL fallback for offline/resilience. |

## Phase 1: Fixes

### 1.1 Faded chips when it's too long

**Problem:** NL keyword extraction can return long labels (e.g., "John Smith International Airport") that are not marked `isTruncated`. Those chips render without the gradient fade and can overflow.

**Fix:** Cap extracted keyword labels in `NaturalLanguageSummarizer`. When the extracted label exceeds a threshold (e.g., 20 chars), truncate and return `SummarizationResult(label: truncated, isTruncated: true)`.

**Files:**
- `GraceNotes/GraceNotes/Services/Summarization/NaturalLanguageSummarizer.swift`
- `GraceNotesTests/Services/Summarization/NaturalLanguageSummarizerTests.swift`

## Phase 2: Chip Delete

### Long-press chip → show "x" → tap "x" to delete

**Behavior:**
1. Long-press a chip → overlay shows "x" button in top-trailing corner.
2. Tap "x" → remove chip; progress count updates (e.g., "4 of 5" → "3 of 5").
3. Completion logic and bottom count stay correct.

**Implementation:**
- **ChipView:** Add optional `onDelete: (() -> Void)?`; long-press gesture shows delete button; tap calls `onDelete`.
- **SequentialSectionView:** Add `onDeleteChip: ((Int) -> Void)?`; pass through to ChipView.
- **JournalScreen:** Wire delete handlers for each section.
- **JournalViewModel:** Add `removeGratitude(at:)`, `removeNeed(at:)`, `removePerson(at:)`; validate index, remove, autosave.

**Files:**
- `GraceNotes/GraceNotes/Features/Journal/Views/ChipView.swift`

## Phase 3: External Summarizer API

**Goal:** Add optional cloud-based summarization for better chip labels, with NL as default and fallback.

**Architecture:** The current `Summarizer` protocol is synchronous. External APIs are async. Introduce `SummarizerAsync` (or extend `Summarizer` with an async method) and update ViewModel add/update flows to be async. `NaturalLanguageSummarizer` can conform via a sync wrapper or a trivial async implementation.

**Implementation outline:**

1. **Protocol:** Add async summarization:
   ```swift
   protocol Summarizer {
       func summarize(_ sentence: String) -> SummarizationResult  // sync, for NL
   }
   protocol SummarizerAsync {
       func summarize(_ sentence: String) async throws -> SummarizationResult
   }
   ```
   Or: single protocol with `func summarize(_ sentence: String) async throws -> SummarizationResult`; `NaturalLanguageSummarizer` implements it with immediate return.

2. **CloudSummarizer:** New type that:
   - Calls OpenAI-compatible API (https://chat.cloudapi.vip via REST)
   - Prompt: "Summarize this into 1–5 words for a chip label: [sentence]"
   - On failure (network, timeout, quota): fall back to `NaturalLanguageSummarizer`
   - API key stored in Keychain

3. **ViewModel:** Add/update methods become `async`; View calls them with `Task { await vm.addGratitude(...) }`.

4. **Settings:** Toggle "Use cloud summarization" + API key entry (or service-specific setup). Persist in UserDefaults/Keychain.

5. **Fallback:** Always fall back to NL on API failure so the app remains usable offline.

**API choice:** Use **chat.cloudapi.vip** (OpenAI-compatible; supports GPT, Claude, Gemini, DeepSeek, etc.). Base URL: https://chat.cloudapi.vip/v1. Document the choice and base URL in code comments.

**Files:**
- `GraceNotes/GraceNotes/Services/Summarization/Summarizer.swift` — extend or add async protocol
- `GraceNotes/GraceNotes/Services/Summarization/NaturalLanguageSummarizer.swift` — conform to async if needed
- `GraceNotes/GraceNotes/Services/Summarization/CloudSummarizer.swift` — new
- `GraceNotes/GraceNotes/Services/Summarization/SummarizerFactory.swift` or similar — select NL vs Cloud based on settings
- `GraceNotes/GraceNotes/Features/Settings/` — new settings view for summarizer preference
- `GraceNotes/GraceNotes/Features/Journal/ViewModels/JournalViewModel.swift` — async add/update
- `GraceNotes/GraceNotes/Features/Journal/Views/JournalScreen.swift` — `Task { await ... }` for add/edit
- `GraceNotes/GraceNotes/Application/GraceNotesApp.swift` — add Settings tab or gear icon
- Tests: MockSummarizer for async; update ViewModel tests

## Phase 4: Chinese Language Support

**Scope:** zh-Hans (Simplified Chinese) only.

**Steps:**
1. Enable Chinese (Simplified) in Xcode project localizations.
2. Create `zh-Hans.lproj/Localizable.strings` (or `Localizable.xcstrings`).
3. Replace hard-coded strings with `String(localized:)` or `Text(LocalizedStringKey(...))`.
4. Add base `en.lproj` strings so English is explicit.

**Strings to localize:** All user-facing text from JournalScreen, HistoryScreen, JournalShareCardView, GraceNotesApp, SequentialSectionView, error messages, tab labels, placeholders, etc. (See grep results from planning for full list.)

## Phase 5: Exploration — Monthly Calendar View

**Deliverable:** `../05-exploration-calendar-view.md` with pros/cons and recommendation.

**Content:**
- **Pros:** Faster visual scanning, familiar metaphor, streak/gap visualization, natural month navigation.
- **Cons:** More complex layout, ambiguous empty vs. no-entry semantics, potential accessibility issues with grid vs. list, higher implementation effort.
- **Recommendation:** Defer implementation to 0.3.0; implement only if analysis supports it.
